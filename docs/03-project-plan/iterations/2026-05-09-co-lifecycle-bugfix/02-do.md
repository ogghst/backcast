**Decisions Made:**

- **Only `change-order-approve`, not `change-order-update`**: The viewer can approve/reject COs assigned by the approval matrix, but cannot create, edit, or delete CO content. This is the minimum permission needed.
- **Two permission sources updated**: The system uses two RBAC layers: (1) `ProjectRole` enum in `app/core/enums.py` for project-level access via `_role_has_permission`, and (2) JSON config in `config/rbac.json` for system-level `RoleChecker` used by the approve/reject endpoints. Both needed updating.
- **Manager and editor do NOT get approve permission**: The approval matrix assigns LOW-impact COs to viewers specifically. Manager and editor retain their existing permission sets.
- **BUG-4: Atomic SLA cleanup via `additional_updates`**: Used the `additional_updates` parameter on `UpdateChangeOrderStatusCommand` instead of a separate `self.update()` call. This creates a single version with both the status transition AND the SLA cleanup, matching the pattern already used by `submit_for_approval()`. The alternative (separate `self.update()` call as done in `reject_change_order()`) would create an extra version unnecessarily.
- **Patch target for `ProjectService`**: New tests patch `app.services.project.ProjectService` instead of `app.services.change_order_service.ProjectService` because `ProjectService` is imported locally inside `merge_change_order()`, not at module level. The existing tests' patch target works due to `patch()` adding and removing the attribute, but only when the attribute was previously set by another test in the same session.
- **BUG-2: `NOT isempty` as defensive check**: PostgreSQL's `@>` operator already excludes empty ranges (empty range contains no timestamps). However, adding `NOT isempty(valid_time)` to the time-travel path ensures consistency with the current-version path, documents intent explicitly, and provides a safety net if the SQL conditions are ever refactored. Tests verify both paths correctly exclude empty-range versions using WBE entity with raw SQL `'empty'::tstzrange` inserts.
- **Empty-range test uses `deleted_at` on zombie version**: The `uq_wbes_wbe_id_branch_current` partial unique index (`WHERE upper(valid_time) IS NULL AND deleted_at IS NULL`) prevents coexistence of two "current" versions. Since `upper('empty'::tstzrange)` returns NULL, an empty-range zombie with `deleted_at IS NULL` would violate this constraint. The test sets `deleted_at` to a future date on the zombie version, which bypasses the constraint and is realistic (residual from soft-delete flows).
