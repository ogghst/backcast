## Execution Summary

All 7 phases of the implementation plan were executed. The unified RBAC system has been implemented with the following components:

### Phase 1: Data Model (BE-001, BE-002, BE-003)

- **BE-001**: Created `UserRoleAssignment` entity (`backend/app/models/domain/user_role_assignment.py`)
  - SimpleEntityBase pattern (non-versioned)
  - ScopeType enum: GLOBAL, PROJECT, CHANGE_ORDER
  - Fields: user_id, role_id, scope_type, scope_id, metadata_ (JSONB), granted_by, granted_at, expires_at
  - UniqueConstraint on (user_id, scope_type, scope_id)
  - Relationships: user, role, granted_by_user (viewonly)

- **BE-002**: Created schemas (`backend/app/models/schemas/user_role_assignment.py`)
  - UserRoleAssignmentCreate (with scope validation)
  - UserRoleAssignmentUpdate
  - UserRoleAssignmentRead (with optional role_name enrichment)
  - UserRoleAssignmentResponse

- **BE-003**: Created Alembic migration (`backend/alembic/versions/20260510_add_user_role_assignments_table.py`)
  - Creates user_role_assignments table with all indexes and constraints
  - Revision: 20260510_unified_rbac, depends on: 20260509_fix_notification_fk

### Phase 2: Service Layer (BE-004 to BE-011)

- **BE-004 to BE-011**: Created `UnifiedRBACService` (`backend/app/core/rbac_unified.py`)
  - Two-tier cache: permissions (1h TTL) + assignments (5min TTL)
  - ContextVar session injection (thread-safe)
  - `has_permission()`: scope resolution (global -> project -> change_order), admin bypass
  - `has_authority_level()`: reads authority_level from metadata, compares hierarchy
  - CRUD: `assign_role()`, `revoke_role()`, `update_assignment()`, `get_user_roles()`, `get_assignments_by_scope()`
  - Cache invalidation on write operations
  - Fail-secure defaults (deny on cache miss/system error)
  - Singleton pattern with `get_unified_rbac_service()` / `set_unified_rbac_service()`

### Phase 3: API Layer (BE-012, BE-023)

- **BE-012**: Created CRUD API routes (`backend/app/api/routes/user_role_assignments.py`)
  - POST / (create), GET / (list), GET /{id}, PUT /{id}, DELETE /{id}
  - Admin-only write access, admin-only read
  - Role name enrichment on read endpoints
  - Registered at `/api/v1/role-assignments`

- **BE-023**: Updated `backend/config/rbac.json`
  - Added `change_order_approver` role with appropriate permissions

### Phase 4: Migration (BE-019, BE-020)

- **BE-019**: Created data migration (`backend/alembic/versions/20260510b_migrate_existing_roles_to_unified_rbac.py`)
  - Migrates User.role -> UserRoleAssignment (scope_type='global')
  - Migrates ProjectMember -> UserRoleAssignment (scope_type='project')
  - Uses INSERT...SELECT with EXISTS checks to avoid duplicates
  - Revision: 20260510_unified_rbac_data, depends on: 20260510_unified_rbac

### Phase 5: Integration (BE-021, BE-022)

- **BE-021**: Updated auth dependencies (`backend/app/api/dependencies/auth.py`)
  - RoleChecker now delegates to UnifiedRBACService first, falls back to legacy RBAC
  - ProjectRoleChecker now delegates to UnifiedRBACService first, falls back to legacy
  - Backward-compatible approach: old checkers still work, new system takes priority

- **BE-022**: Updated ChangeOrderWorkflowService (`backend/app/services/change_order_workflow_service.py`)
  - Approval and rejection now use UnifiedRBACService for permission checks
  - Falls back to ApprovalMatrixService for detailed error messages
  - Uses ContextVar session injection pattern

### Phase 6: Testing (BE-013 to BE-018, BE-024, BE-025)

- Created 50 new tests across 3 test files:
  - `tests/unit/core/test_rbac_unified.py` (27 tests): cache, permissions, authority, CRUD, singleton
  - `tests/unit/models/test_user_role_assignment.py` (13 tests): entity creation, fields, repr
  - `tests/unit/models/test_user_role_assignment_schemas.py` (10 tests): validation, serialization

- All 115 tests pass (50 new + 65 existing RBAC tests)
