## Technical Decisions Made

### 1. Abstract Base Class Pattern

**Decision:** Use ABC instead of Protocol for RBAC service interface.

**Reasoning:**

- Explicit inheritance makes the contract clearer
- Easier to add future helper methods
- Consistent with existing service patterns in the project
- Better IDE support for implementation checking

**Alternatives Considered:**

- Using Protocol (structural subtyping) - rejected for clarity
- Direct implementation - rejected for extensibility

### 2. OR Logic for Combined Authorization

**Decision:** When both `allowed_roles` and `required_permission` are provided, grant access if EITHER condition is met.

**Reasoning:**

- More flexible and less restrictive
- Allows for "admin bypass" patterns (admin role OR specific permission)
- Matches common authorization patterns in real-world applications

**Impact:** Documented clearly in docstrings to avoid confusion

### 3. Global Singleton for RBAC Service

**Decision:** Use module-level singleton with dependency injection

**Reasoning:**

- Avoids reloading JSON config on every request
- Testable via `set_rbac_service()` for dependency override
- Follows FastAPI dependency injection patterns
- Allows for future database-backend swap without changing route code

### 4. Type Annotations for JSON Returns

**Decision:** Added explicit type annotations for `json.load()` returns

**Reasoning:**

- MyPy strict mode requires explicit annotation for Any returns
- Improves type safety and IDE autocomplete
- Pattern:
  ```python
  config: dict[str, Any] = json.load(f)
  permissions: list[str] = role_config.get("permissions", [])
  ```
