## Bug Summary

When updating a cost element forecast with a specific control date (e.g., Feb 1, 2026), the new version's `valid_time` should start from that control date. However, the current behavior incorrectly uses the current time as the `valid_time` start.

**Root Cause:** The cost element forecast update endpoint (`PUT /api/v1/cost-elements/{cost_element_id}/forecast`) does not accept or pass the `control_date` parameter from the client.

**Expected Behavior:**
- **Valid Time**: When the fact was true in the real world (user-specified control date)
- **Transaction Time**: When the fact was recorded in the database (current system time)

When a user provides `control_date=2026-02-01T00:00:00Z`:
- The new version's `valid_time` should start at 2026-02-01 (user-specified)
- The `transaction_time` should reflect when the record was actually created (current system time)

## Implementation Notes

### Bitemporal Semantics

The fix correctly implements bitemporal semantics as defined in `docs/02-architecture/cross-cutting/temporal-query-reference.md`:

- **Valid Time** (`valid_time`): When the fact was true in the real world (user-specified via `control_date`)
- **Transaction Time** (`transaction_time`): When the fact was recorded in the database (current system time)

When a user provides `control_date=2026-02-01T00:00:00Z`:
- The new version's `valid_time` starts at 2026-02-01 (user-specified)
- The `transaction_time` starts at the current system time (when the record was created)

### API Usage Example

```bash
# Update forecast with specific control date
curl -X PUT "http://localhost:8000/api/v1/cost-elements/{cost_element_id}/forecast?control_date=2026-02-01T00:00:00Z" \
  -H "Authorization: Bearer {token}" \
  -H "Content-Type: application/json" \
  -d '{
    "eac_amount": "120000.00",
    "basis_of_estimate": "Updated forecast with control date"
  }'
```

### Service Layer Integration

The fix leverages existing service layer methods that already support `control_date`:
- `ForecastService.update(control_date=...)`
- `ForecastService.create_for_cost_element(control_date=...)`

These methods use the `CreateVersionCommand` and `UpdateVersionCommand` which properly set the `valid_time` based on the `control_date` parameter.
