#### 🟢 GREEN Cycle 1: `_compare_kpis()` Implementation

**Method Implemented:** `ImpactAnalysisService._compare_kpis()`

**Implementation Details:**

- Pure function taking main/change values for BAC, budget_total, gross_margin
- Calculates delta = change - main
- Calculates delta_percent:
  - Returns 0.0 if delta is 0
  - Returns None if main is 0 (avoid division by zero)
  - Returns (delta / main * 100) otherwise
- Returns `KPIScorecard` with all three KPIs populated

**Tests Updated:**

- `test_compare_kpis_no_changes` - Now calls service method and asserts results
- `test_compare_kpis_happy_path` - Verifies delta and percent calculations

**Status:** ✅ Implemented and tests passing

#### 🟢 GREEN Cycle 2: `_build_waterfall()` Implementation

**Method Implemented:** `ImpactAnalysisService._build_waterfall()`

**Implementation Details:**

- Pure function taking current_margin and margin_delta
- Calculates new_margin = current_margin + margin_delta
- Returns 3 WaterfallSegment objects:
  1. Current Margin (baseline, is_delta=False)
  2. Change Impact (delta, is_delta=True)
  3. New Margin (result, is_delta=False)

**Status:** ✅ Implemented and tests passing

#### 🟢 GREEN Cycle 3: `_compare_entities()` Implementation

**Methods Implemented:**

- `ImpactAnalysisService._compare_entities()` - Main comparison method
- `ImpactAnalysisService._compare_wbe_lists()` - WBE comparison logic
- `ImpactAnalysisService._compare_cost_element_lists()` - Cost Element comparison logic

**Implementation Details:**

- Uses branch-aware queries with STRICT mode
- Gets current versions from main branch and change branch
- Compares by root_id to identify:
  - **Added**: Entity in change branch but not in main
  - **Removed**: Entity in main but not in change branch (negative budget impact)
  - **Modified**: Entity in both with different financial values
- Only compares financial fields (budget_allocation for WBEs, budget_amount for Cost Elements)

**Schema Updates:**

- Changed `branch_id: int` to `branch_name: str` to match existing patterns
- Updated `ImpactAnalysisResponse` to use `change_order_id: UUID` and branch names as strings
- Fixed `EntityChangeType` to use `TypeAlias` instead of subclassing `Literal` (Python limitation)

**Status:** ✅ Implemented and tests passing

#### 🟢 GREEN Cycle 4: `_generate_time_series()` Implementation

**Method Implemented:** `ImpactAnalysisService._generate_time_series()`

**Implementation Details:**

- Queries total budget from main branch using `func.sum(WBE.budget_allocation)`
- Queries total budget from change branch
- Returns list of `TimeSeriesPoint` objects (one point for current month)
- Simplified implementation (single time point) - full weekly historical tracking requires additional data model

**Status:** ✅ Implemented and tests passing

#### 🟢 GREEN Cycle 5: `analyze_impact()` Main Orchestration

**Method Implemented:** `ImpactAnalysisService.analyze_impact()`

**Implementation Details:**

- Queries Change Order to retrieve `project_id`
- Calculates KPIs (BAC, budget_total, gross_margin) from both branches
- Gross margin simplified to 20% of budget (production would use actual cost/revenue)
- Calls all comparison methods:
  - `_compare_kpis()` for KPI scorecard
  - `_compare_entities()` for entity changes
  - `_build_waterfall()` for waterfall chart
  - `_generate_time_series()` for S-curve data
- Returns complete `ImpactAnalysisResponse`

**Status:** ✅ Implemented and tests passing
