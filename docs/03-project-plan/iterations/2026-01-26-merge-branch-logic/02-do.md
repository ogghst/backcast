## Decisions Made

### BE-001 Design Decisions

**Decision 1: Service Interface**
- **Choice:** Create three separate methods (`discover_wbes`, `discover_cost_elements`, `discover_projects`)
- **Rationale:** Follows single responsibility principle, allows independent querying per entity type
- **Impact:** More explicit than a generic `discover_entities(entity_type)` method

**Decision 2: Query Filtering**
- **Choice:** Filter by `branch == branch_name` AND `deleted_at IS NULL`
- **Rationale:** Standard pattern for active entities in branching system
- **Impact:** Aligns with existing repository patterns

**Decision 3: Async/Await Pattern**
- **Choice:** Use async/await for all database operations
- **Rationale:** Consistent with FastAPI and SQLAlchemy 2.0 async patterns
- **Impact:** Maintains consistency with existing service layer

**Decision 4: Type Annotations (REFACTOR)**
- **Choice:** Remove explicit `Select[T]` type hints, let SQLAlchemy infer types
- **Rationale:** MyPy strict mode incompatibility with SQLAlchemy 2.0 select() return types
- **Impact:** Code remains type-safe, passes MyPy strict mode

**Decision 5: Project Entity Exclusion (BE-003)**
- **Choice:** Exclude Projects from merge orchestration
- **Rationale:** Project is TemporalService, not BranchableService - lacks merge_branch method
- **Impact:** Only WBEs and CostElements are merged (correct behavior for change orders)

**Decision 6: Integration Test Pattern (BE-005)**
- **Choice:** Create entities on main branch first, then source branch, then merge
- **Rationale:** MergeBranchCommand requires target version to exist
- **Impact:** Tests accurately reflect real-world merge workflow
