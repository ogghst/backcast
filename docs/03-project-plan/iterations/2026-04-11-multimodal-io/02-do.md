**Decisions Made:**

1. **Image rendering via data URLs in FilePreview**
   - Decision: Use `data:${file_type};base64,${content}` for img src
   - Rationale: Backend stores raw base64 (without data URL prefix); frontend constructs the full data URL
   - Impact: No network requests needed for image display; content is inline

2. **Images merged into attachments array (no separate `images` field)**
   - Decision: Both images and documents go into the `attachments` array in WSChatRequest
   - Rationale: Backend already handles image attachments via `format_multimodal_messages`; separate `images` field is redundant
   - Impact: Simplified upload flow; `images` field on WSChatRequest remains for backward compat but is not populated

3. **No MarkdownRenderer schema change for data URLs**
   - Decision: Keep existing sanitization (block data: URLs in markdown images)
   - Rationale: react-markdown strips data URLs from markdown image syntax anyway; inline images use FilePreview component directly
   - Impact: Security posture unchanged; no new attack surface

4. **FilePreview no longer wraps in `<a>` link**
   - Decision: Changed from `<a href={url}>` to `<div>` wrapper
   - Rationale: No URL to link to since files are stored inline; download links would require a separate endpoint
   - Impact: Attachments display as info cards (icon + filename + size) without click-to-open

**Decisions Made:**

1. **Upload Endpoint Architecture**
   - **Decision:** Keep existing upload endpoints unchanged
   - **Rationale:** Upload endpoints return file metadata; attachments are created when messages are created
   - **Impact:** Frontend uploads files first, gets file_id/URL, then includes metadata when sending message
   - **Flow:**
     1. Frontend: `POST /upload-image` → gets `{file_id, url, filename, content_type, size}`
     2. Frontend: Send message with attachment metadata in `message_metadata.attachments`
     3. Backend: `add_message()` creates message with metadata
     4. Backend: `AIConversationAttachment` records created via relationship

2. **Vision Message Format**
   - **Decision:** Use OpenAI's content array format
   - **Rationale:** Standard format for GPT-4 Vision and compatible models

3. **Non-Image Attachments**
   - **Decision:** Mention non-image attachments in text rather than attempting to process them
   - **Rationale:** Vision models can't process PDFs/CSVs
   - **Implementation:** Append `[User attached: filename1.pdf, filename2.csv]` to text content
