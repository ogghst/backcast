**Technical Decisions Made:**

1. **Gaussian S-Curve Implementation**: Used `math.erf` (error function) with scale factor of 3.0 for standard S-curve. Explicit boundary clamping at start/end since erf asymptotically approaches but never reaches 0/1.

2. **Progression Type Storage**: Used PostgreSQL ENUM (`progression_type`) for type safety at DB level. Values: LINEAR, GAUSSIAN, LOGARITHMIC.

3. **PV Calculation Endpoint**: Created dedicated `/pv` endpoint that calculates Planned Value dynamically based on:
   - Current date (query parameter)
   - BAC - Budget at Completion (query parameter)
   - Baseline's progression type and date range

4. **Service Method Naming**: Overrode `get_by_id(root_id, branch)` to match parent class pattern rather than using entity-specific name, maintaining consistency with BranchableService.

5. **API Pagination**: Implemented direct SQLAlchemy queries instead of using non-existent `service.list()` method (pre-existing issue in forecasts API).

**Deviations from Plan:**

- **Integration Tests**: Skipped integration tests for service layer due to pre-existing database migration issue affecting all tests (foreign key constraint error on `departments` table). Unit tests verify core logic.
- **PV Calculation Tests**: Created unit tests for PV calculation but couldn't run due to database fixture issue. Logic is verified through progression tests.

**Blockers and Challenges:**

1. **Database Migration Issue**: Pre-existing foreign key constraint error in test database setup affecting all tests. Not introduced by this implementation.
   - Error: `InvalidForeignKeyError: there is no unique constraint matching given keys for referenced table "departments"`
