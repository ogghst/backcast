**Technical Decisions Made:**

- **Decision:** Use `BranchPublic` schema for API response.
  - **Reasoning:** Provides Strong typing and clear contract for frontend. Allows including metadata like CO status easily.
- **Decision:** Integrate `useProjectBranches` into existing `useProjects` hook file.
  - **Reasoning:** Keeps project-related API hooks consolidated.
- **Decision:** Create `ProjectBranchSelector` smart component.
  - **Reasoning:** Decouples the presentation logic (`BranchSelector`) from the data fetching and state management (`useTimeMachineStore`), making the header `TimeMachineCompact` cleaner.

**Deviations from Plan:**

- Initially thought about a separate service file for frontend branch fetching, but decided to stick to the pattern of grouping hooks by feature (`useProjects.ts`).

**Blockers and Challenges:**

- **Challenge:** Frontend `Branch` type vs `BranchOption` type mismatch.
  - **Resolution:** Mapped API response to `BranchOption` format within the `ProjectBranchSelector` component.

**Integration Points:**

- **Backend:** `ProjectService` integrates with `ChangeOrder` model.
- **Frontend:** `TimeMachineCompact` integrates `ProjectBranchSelector`.
