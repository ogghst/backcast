## Decisions Made

### Decision 1: Type Definitions Without Generated Client
**Context:** Backend is not running, cannot regenerate OpenAPI client
**Options:**
1. Wait for backend to be running
2. Manually create type definitions matching backend schemas
3. Use `any` types temporarily

**Decision:** Option 2 - Create manual type definitions
**Rationale:** Frontend development can proceed independently. Type definitions are stable and documented in the backend plan.
**Impact:** Will need to verify types match once backend is running. Low risk since schemas are well-defined.

### Decision 2: Use Native Fetch Instead of Generated Client
**Context:** OpenAPI client doesn't have AI endpoints yet
**Options:**
1. Use generated client (not available)
2. Use native fetch with manual types
3. Use axios directly

**Decision:** Option 2 - Native fetch
**Rationale:** Simple, no additional dependencies. Can easily swap to generated client later.
**Impact:** Code will need to be updated to use generated client once available. Isolated in API layer.

#### FE-025: Fix AI Providers and Assistants Not Showing Inactive Records

| Cycle | Test Name | RED Reason | GREEN Implementation | REFACTOR Notes |
|-------|-----------|------------|---------------------|----------------|
| 1 | Database contains 3 providers but only 1 shown | Backend API filters inactive records by default | Added `includeInactive` parameter to frontend hooks | Backend endpoint has `include_inactive: bool = False` |
| 2 | useAIAssistants doesn't support includeInactive | Hook signature doesn't accept parameter | Updated hook to accept `includeInactive?: boolean` | Matches useAIProviders pattern |
| 3 | API call doesn't pass include_inactive query param | No params passed to axios | Added `params = includeInactive ? { include_inactive: "true" } : {}` | Follows same pattern as useAIProviders |
| 4 | Query key doesn't include includeInactive | Cache key is static | Updated queryKeys.ai.assistants.list to accept parameter | Matches queryKeys.ai.providers.list pattern |
| 5 | AIAssistantList doesn't pass includeInactive | Component calls hook without arguments | Updated to call `useAIAssistants(true)` | Ensures all records shown |
| 6 | AIProviderList doesn't pass includeInactive | Component calls hook without arguments | Updated to call `useAIProviders(true)` | Ensures all records shown |

**Root Cause Analysis:**
- Backend API endpoints (`/api/v1/ai/config/providers` and `/api/v1/ai/config/assistants`) have `include_inactive: bool = False` as default parameter
- Frontend hooks were calling these endpoints without passing `include_inactive=true`
- Result: Only active records (is_active=true) were being returned by the API
- Database verification showed 3 providers and 2 assistants, but UI was showing fewer

**Status:** COMPLETED
**Files Modified:**
- `frontend/src/api/queryKeys.ts` - Updated `assistants.list` to accept `includeInactive` parameter
- `frontend/src/features/ai/api/useAIAssistants.ts` - Added `includeInactive` parameter to hook and API call
- `frontend/src/features/ai/components/AIAssistantList.tsx` - Updated to call `useAIAssistants(true)`
- `frontend/src/features/ai/components/AIProviderList.tsx` - Updated to call `useAIProviders(true)`

**Testing Approach:**
