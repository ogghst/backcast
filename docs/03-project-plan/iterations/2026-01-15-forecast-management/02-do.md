## Technical Decisions Made

### Decision 1: Forecast is Branchable (not just Versionable)

**Reasoning:**
- The plan explicitly states: "Forecasts are **branchable** to enable 'What-if' scenarios"
- Change Orders need to carry specific EACs without affecting main branch
- Supports simulation of different cost scenarios

**Implementation:**
- `Forecast(EntityBase, VersionableMixin, BranchableMixin)`
- `ForecastService(BranchableService[Forecast])`

**Alternatives Considered:**
- Could have made forecasts versionable only (like CostRegistration)
- Rejected because "What-if" scenarios are a key requirement

### Decision 2: Field Naming - `basis_of_estimate`

**Reasoning:**
- Plan specifies: "`reason` field renamed to `basis_of_estimate`"
- More descriptive and domain-appropriate terminology
- Aligns with EVM industry terminology

**Implementation:**
- `basis_of_estimate: Mapped[str]` (required Text field)
- Included in both domain model and schemas

### Decision 3: Financial Field Precision

**Reasoning:**
- Following existing patterns from CostRegistration and CostElement
- Standard precision for currency calculations

**Implementation:**
```python
eac_amount: Mapped[Decimal] = mapped_column(
    Numeric(precision=15, scale=2), nullable=False
)
```

### Decision 4: Test Fixture Design

**Reasoning:**
- Tests need a full hierarchy: Department -> CostElementType -> Project -> WBE -> CostElement
- Re-used existing test patterns from `test_cost_element_service.py`

**Implementation:**
- Created `cost_element_setup` fixture that builds full hierarchy
- Uses existing services (DepartmentService, CostElementTypeService, etc.)

## Deviations from Plan

### Deviation 1: Test File Structure

**What changed:**
- Originally planned to put imports inside test function (RED phase pattern)
- Changed to standard imports at module level after creating implementation files

**Why it changed:**
- Standard Python practice
- Required for test discovery
- Cleaner code organization

**Impact:** None - test structure is standard

### Deviation 2: No Foreign Key Constraint on cost_element_id

**What changed:**
- Database migration does NOT include FK constraint to cost_elements

**Why it changed:**
- Following existing pattern from cost_elements table
- In bitemporal system, root IDs appear in multiple rows (versions)
- FK constraints require UNIQUE, which root IDs cannot have
- Referential integrity enforced at application level

**Impact:** None - follows established pattern
