**Decisions Made:**

- Used `JsonRBACService` with `Path` resolved to real `config/rbac.json` (not MockRBACService): Tests verify actual config file state, catching deployment issues early
- Each test verifies both positive (has permission) and negative (lacks permission) cases in a single method: Keeps tests cohesive and readable without unnecessary fragmentation
- Negative permissions in tests cover the most dangerous cross-role violations (e.g., `ai-viewer` lacks `project-create`, `ai-manager` lacks `project-delete`, `ai-admin` lacks `wbe-create`)

- Used `Any` type for contextvar session (not `AsyncSession`): `rbac.py` intentionally avoids importing SQLAlchemy; callers handle the session lifecycle
- `self.session` retains priority over contextvar fallback: backward-compatible with all existing callers that mutate `rbac_service.session` directly
- `_role_has_permission()` uses `app.core.enums.ProjectRole` enum values (e.g., `"project_viewer"` not `"viewer"`): test mock must use valid enum string values

- `filter_tools_by_role()` uses lazy import of `get_rbac_service` (inside function body): matches existing `filter_tools_by_execution_mode()` pattern; allows patching at `app.core.rbac.get_rbac_service` in tests
- `assistant_role` is `None` by default on `create_agent()`: backward-compatible -- when no role is specified, no filtering occurs (all tools pass through)
- Role filtering applied AFTER execution mode filtering and BEFORE middleware/subagent construction: ensures subagents also receive only role-permitted tools
- Patching `app.core.rbac.get_rbac_service` (not `app.ai.tools.get_rbac_service`): required because `filter_tools_by_role` uses a local import from `app.core.rbac`, not a module-level import

- BE-003 removed 12 lines from decorator.py (the `from app.core.rbac import get_rbac_service` through the permission loop): the decorator no longer imports or calls any RBAC service
- Removed 7 obsolete tests that asserted decorator-level permission enforcement: these tested behavior that is now exclusively the middleware's responsibility
- Tests use real `ToolContext` instances (not `MagicMock(spec=ToolContext)`) with `ToolSessionManager` patched: avoids `_root_session` serialization errors when LangChain tries `model_dump()` on the context
- `test_middleware_uses_contextvar_session` now un-skipped and passing after BE-005: patches `set_rbac_session` at module level to verify it is called with `ctx.session`, and asserts `rbac_service.session` is never mutated
- BE-005 changes are purely caller-side: no changes to `rbac.py` itself (contextvar helpers added in BE-004 remain untouched); `inject_rbac_session()` preserved for other callers
- Removed `hasattr(rbac_service, "session")` guard from both callers: contextvar accepts any value, and `JsonRBACService` already uses `self.session or get_rbac_session()` fallback internally
- Removed try/finally session swap from middleware: contextvar is task-scoped by default, so no restoration is needed -- each async task gets its own context

- TEST-002 integration tests use `real_rbac_service` fixture with actual `JsonRBACService(config_path=...)` and `all_tools` fixture that clears tool cache and creates real tools via `create_project_tools()`: verifies end-to-end that `filter_tools_by_role()` works with real RBAC config and real tool permission metadata
- Integration tests assert both positive (tool present) and negative (tool absent) cases with named tool lists: catches any mismatch between rbac.json permissions and tool permission metadata
- `test_filter_tools_by_role_combined_with_execution_mode` verifies the intersection of role filter + SAFE execution mode: confirms both filters compose correctly without interference
- Tool names in assertions matched against actual tool registry output (not guessed): initial failure on `update_change_order` revealed the actual tool is `submit_change_order_for_approval` with `change-order-update` permission
